# BLACKLETTER — THE ORDER OF OPERATIONS

*How this system gets built. The rule that governs every other rule.*

*Version 1 — captured from a decision made in a build session. This one comes first because it governs the writing of all the others.*

---

## THE RULE

**No action begins before the rule.**

The rule that governs a thing is written down first. Then the thing is built to that rule. Never the other way around.

- Not build, then justify.
- Not act, then notice the principle afterward.
- The rule leads. The action follows and conforms.

If the rule is not written, the build does not start. The rule gets written, then the build proceeds to it.

---

## WHY THIS RULE EXISTS

For a long time this system was built the other way — action first, rule discovered afterward. That produced real progress and real understanding, but it also produced confusion: the same lessons had to be re-learned, the same distinctions re-derived, because the rules lived only in memory and in conversations that scrolled away.

The confusion was not wasted — it is where the rules came from. You cannot know the right order until you have built the wrong version first. But now that the order is known, there is no reason to keep paying that price. From here forward: rule first, then build.

This rule is itself proof of the pattern. It could not have been written on day one. It is a reflection on having done it the hard way. That is exactly why it belongs at the top — it is the most earned rule in the system.

---

## WHAT THIS RULE PRODUCES

When every action is preceded by its rule, the whole system becomes traceable in the same way its definitions are traceable:

- Every artifact traces back to a stated rule.
- Every rule traces back to a source or a recorded decision.
- No code exists without a rule behind it. No data structure exists without a decision that shaped it.

The gate demands provenance for **meaning**. This rule demands provenance for **building**. Same discipline, turned on the process itself.

It also means the artifacts can never quietly drift from the principles — because the principle precedes every artifact, and every artifact is built to a written rule. There is no gap for drift to enter.

---

## THE COMPANION RULES (build discipline)

These are the working rules that put "no action before the rule" into practice. They were learned by building the wrong version first; they are written here so they never have to be re-learned.

### Capture before it scrolls away.
The sharpest thinking happens in conversation and then is lost. A decision that lives only in a chat or a phone note is not captured — it is at risk. When a rule is decided, it gets written into an owned document that same session. The discipline lives in the artifacts and in the person, never in any single conversation, which does not persist.

### Hand-curation is clean; bulk extraction is poison.
Every clean data source in this project was built or verified by hand. Every corrupted one came from an automated bulk pull. This is not a coincidence — it is a law. Bulk-extracted data is treated as suspect until verified, never trusted on arrival. (Learned twice: the scrapped AI etymology batches, and the poisoned definition databases.)

### Empty is shown as empty.
A field with no verified source reads as empty — openly, not as a blank that could be mistaken for filled. Honesty about what is missing is worth more than the appearance of completeness. A beautiful, full-looking artifact built on unverified filling is the exact trap this project exists to refuse.

### Solve, never supply.
When reconstructing damaged data: if the correct answer is forced by evidence already present, recover it — that is solving. If recovering it requires reaching outside for a plausible answer, stop — that is supplying, and supplying is fabrication. A rule that cannot prove its target does not fire.

### The rule proposes; the source disposes.
Any automated rule (rerooting, cleanup, classification) only acts where the source confirms it. A rewrite that cannot find its target in the real source simply does not happen. This protects against the rule mangling a case it merely looked like it fit. The rule's reasoning is always shown, so a misfire can be seen and caught.

### One function, one place.
(From the Architecture Map, repeated here because it is also a build rule.) Every part does one job and lives in one place. When building anything new, the first test is: does this violate the One Rule, or collapse any of the four parts? If yes, it is rejected before it is built.

---

## THE ORDER, STATED PLAINLY

1. Meet the problem.
2. Decide the rule that answers it.
3. Write the rule down.
4. Build the thing to the rule.
5. Check the thing against the rule.

Steps 2 and 3 never get skipped to reach step 4 faster. The writing is not overhead — it is the part that makes the building real.
